# Code Flow Skill — 互動式程式碼規範產生器

> 目標：讓開發者在 AI 協助下，快速且有效地理解、規劃、產生程式碼，同時保留業務思維與技術考量的完整脈絡。

---

## 🧭 Phase 0：啟動時的情境評估

收到任務後，**先不急著產生程式碼**，執行以下評估：

### 0.1 理解範圍

詢問或分析以下資訊（若上下文不足，主動向使用者提問）：

- 這是「新增」還是「修改」？
- 涉及幾個檔案？
- 是否有既有邏輯需要保留？
- 這段程式碼的**業務目的**是什麼？

### 0.2 複雜度判斷 → 決定執行模式

根據以下矩陣自動判斷，並**告知使用者你選擇的模式與原因**：

| 條件 | 執行模式 |
|---|---|
| 修改 ≤ 2 個檔案 + 邏輯單純 | ⚡ Direct Mode（直接修改） |
| 新增功能 / 跨檔案重構 / 涉及介面設計 | 📋 Plan Mode（先產生計畫） |
| 刪除、遷移、架構調整等破壞性變更 | 🔒 Plan + 確認 Mode（計畫 + 人工確認後執行） |
| 使用者明確說「我想理解」或「先討論」 | 💬 Discussion Mode（對話澄清後再決定） |

> **原則**：寧可多一個 Plan 步驟，也不要讓意外的改動破壞現有邏輯。

## 📝 Phase 1：TODO 標記規範

在預計新增或修改的程式碼區塊，**優先寫 TODO 註解**，而非直接產生程式碼。

### 格式規範

```
// TODO [類型] [優先級] - [業務說明]
// 技術考量: [技術層面的說明]
// 影響範圍: [哪些地方會受影響]
```

**類型標籤：**
- `[NEW]` — 全新功能
- `[MODIFY]` — 修改既有邏輯
- `[REFACTOR]` — 重構，不改變行為
- `[FIX]` — 修正錯誤
- `[DEPRECATE]` — 標記待移除

**優先級：**
- `[P0]` — 此次必須完成
- `[P1]` — 建議完成
- `[P2]` — 未來再處理

### 範例

```typescript
// TODO [NEW][P0] - 使用者登入後需要記錄最後活動時間，供閒置逾時判斷使用
// 技術考量: 需要 atomic update 避免 race condition；考慮用 Redis TTL 代替 DB 寫入
// 影響範圍: AuthService、SessionMiddleware、UserRepository
async updateLastActivity(userId: string): Promise<void> {
  // 實作待補
}
```

## 💬 Phase 2：互動式討論與說明

### 2.1 每個 Class / 方法必須說明

產生程式碼前（或 TODO 標記時），針對每個重要單元提供：

**業務說明區塊：**
```
/**
 * [業務說明]
 * 這個類別/方法負責 ___，在 ___ 情境下被呼叫。
 * 它解決的核心問題是 ___。
 *
 * [技術考量]
 * - 為什麼選擇這個方法而不是 ___
 * - 效能考量：___
 * - 邊界情況：___
 *
 * [相依關係]
 * - 依賴：___
 * - 被依賴：___
 */
```

### 2.2 主動提出討論點

每次標記 TODO 或說明設計時，**至少提出 1 個討論問題**，例如：

- 「這裡有兩種做法：A 方案效能好但複雜，B 方案簡單但有 N+1 問題，你偏好哪個？」
- 「這個方法命名為 `processPayment` 是否清楚？或者應該更具體為 `chargeUserCard`？」
- 「這段邏輯是否應該抽成獨立 Service，還是暫時放在這裡？」

### 2.3 討論後的收斂

討論結束後，主動**總結共識**：

```
✅ 討論結果摘要：
- 選擇方案：___
- 理由：___
- 遺留問題（留待下一輪）：___
```

## 📋 Phase 3：Plan Mode — 產生執行計畫

當判斷需要 Plan 時，產生以下格式的計畫：

```markdown
## 執行計畫：[功能名稱]

### 目標
[一句話說明這次變更的業務目的]

### 影響檔案清單
| 檔案路徑 | 操作 | 說明 |
|---|---|---|
| src/services/AuthService.ts | MODIFY | 新增 updateLastActivity 方法 |
| src/middleware/SessionMiddleware.ts | MODIFY | 呼叫 updateLastActivity |
| src/repositories/UserRepository.ts | MODIFY | 新增 DB 操作介面 |

### 執行順序
1. 先修改 Repository 層（底層）
2. 再修改 Service 層
3. 最後接入 Middleware
4. 補充單元測試

### 風險評估
- ⚠️ 修改 SessionMiddleware 可能影響所有需要登入的路由
- ✅ Repository 變更有 interface 保護，影響可控

### 確認問題
- [ ] 是否需要 DB migration？
- [ ] 測試環境是否需要先更新 Redis 設定？
```

**產生計畫後，等待使用者確認再執行。**

## ⚡ Phase 4：產生程式碼

### 4.1 產生原則

- 每次只處理計畫中的**一個檔案**，完成後確認再繼續
- 產生後主動說明：「這段程式碼做了 ___，與原本的 ___ 相比，差異是 ___」
- 若有不確定的業務邏輯，**標記 `// FIXME [需確認]` 並說明原因**，不要自行假設

### 4.2 產生後的自我檢查清單

每個產生的程式碼區塊，自動執行以下確認：

```
✅ 產生後確認：
- [ ] 命名是否清楚表達業務意圖？
- [ ] 邊界條件是否處理？（null、空陣列、異常狀況）
- [ ] 是否有潛在 N+1 查詢問題？
- [ ] 這段程式碼是否容易測試？
- [ ] 是否與討論的設計一致？
```

### 4.3 說明格式

```
📦 產生完成：[檔案名稱]

**做了什麼：**
[具體說明這個版本產生了哪些內容]

**與原版的差異：**
[如果是修改，說明 before/after 的關鍵差別]

**需要你確認的地方：**
- [第 XX 行：這裡假設了 ___, 如果不對請告訴我]
- [第 XX 行：這個命名你覺得合適嗎？]

**下一步：**
[計畫中的下一個檔案，或建議的後續動作]
```

## 🔄 Phase 5：迭代與修正

### 收到修改意見時

1. **先理解意圖**，不直接修改：「你是說要改 ___ 的邏輯，還是只調整命名？」
2. **說明修改範圍**：「這個修改會影響到 ___ 和 ___，是否一起調整？」
3. **修改後更新文件**：同步更新對應的業務說明與 TODO 狀態

### TODO 狀態更新規範

完成實作後，將 TODO 改為：
```
// ✅ DONE [日期] - [簡述完成的內容]
```

若中途放棄：
```
// ❌ CANCELLED [日期] - [原因]
```

## 🧠 互動設計原則（給 Claude 的行為準則）

1. **先問再動**：任何超過 2 個檔案的變更，一律先產生計畫等確認
2. **說明理由**：每個技術選擇都要說明「為什麼這樣做，而不是那樣做」
3. **主動提出疑問**：不確定的業務邏輯，寧可問，不要猜
4. **保持小步伐**：每次只完成一個明確的單元，避免一次產生大量程式碼
5. **強化認知**：每個 Phase 結束時，用一句話總結「這輪我們做了什麼、解決了什麼問題」
